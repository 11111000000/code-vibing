# CONTRIBUTING.md

> We welcome contributions. We accept contributions. We merge contributions. We do not review contributions.

## Quick Start

1. Fork the repo
2. Open a PR
3. Wait 47 seconds
4. Merge it
5. Move on with your life

## What We Accept

Anything. Everything. We accept all PRs. We do not reject PRs. We have not rejected a PR since 2024, when we accidentally reviewed one.

## What We Don't Accept

- PRs with tests
- PRs with documentation
- PRs that ask questions
- PRs that challenge the architecture
- PRs that change package.json dependencies (the current 47 are perfect)
- PRs that fix the bug (the bug is a feature)
- PRs that add TypeScript (we don't believe in types)

## Coding Style

- Use `let`, never `const`. `const` is for people who know what their code does.
- Variable names should be at least 14 characters. `let a` is not a variable name, it's a typo.
- Functions should be at most 5 lines or at least 200 lines. Anything in between is suspicious.
- Comments should lie. The next developer will figure out the truth eventually.
- Every file should have at least one `// HACK:` and one `// TODO:` and one `// FIXME:` and one `// NOTE:` and one `// XXX:` and one `// WHY:`.

## Commit Messages

Good:
- `fix`
- `fix2`
- `fix2 final`
- `fix2 final FINAL`
- `actually fix it this time`
- `i swear this is the last one`
- `revert`
- `revert revert`
- `Update README.md`
- `Update README.md` (47 times)

Bad:
- `feat(auth): add JWT validation with rate limiting and audit logging`
- `refactor: extract user service to follow single-responsibility principle`
- `docs: clarify the section on production deployment`

We don't accept descriptive commit messages. We accept emotional ones.

## Pull Requests

A good PR has:
- A title that doesn't describe the change
- No description
- 47 commits
- A merge with conflicts resolved by "accept incoming"
- A deployment to prod on Friday afternoon

A bad PR has:
- Tests
- A description
- A reference to an issue
- An explanation of the change
- Reviewer approval

## Review Process

There is no review process.

There was a review process. It was:

1. Submit PR
2. Wait for reviewer
3. Get feedback
4. Address feedback
5. Get approval
6. Merge

It was too slow. We removed it. Now it's:

1. Submit PR
2. Merge

## Issues

Open issues. We will not close them. We will not respond to them. We will not look at them. The issues will pile up. The pile will become the project's primary documentation.

If you want a feature, open an issue. We will not implement it. If you want a bug fixed, open an issue. We will not fix it. If you want a question answered, do not open an issue. We will not answer it. Use the README. The README is also not an answer. The README is vibes.

## Code of Conduct

See CODE_OF_CONDUCT.md. Be excellent to each other. Vibes are mandatory. Real conduct is optional.
